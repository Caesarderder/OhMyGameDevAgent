# 知识地图更新工作流

当用户调用 `/harness-km-update` 时，执行以下工作流。

---

## 触发条件

用户输入包含 `/harness-km-update` 命令。

---

## 工作流程

### 1. 加载 Harness Skill

读取以下文档到上下文：
- `~/.claude/skills/harness/SKILL.md`（核心概念）
- `~/.claude/skills/harness/references/` 下的所有参考文档

### 2. 分析项目结构

扫描项目以了解：
- 编程语言和框架（src/ 下的子项目）
- 现有的 AGENTS.md 和 ARCHITEACHTURE.md 内容
- docs/ 目录结构和现有文档
- package.json 中的依赖

### 3. 读取现有文档

检查以下文件状态：
- `AGENTS.md` - 是否为空或已有内容
- `ARCHITEACHTURE.md` - 是否为空或已有内容
- `docs/*.md` - 现有文档列表

### 4. 执行知识地图操作

根据用户描述执行 CRUD：

**C (Create) - 创建/新增：**
- 新增 harness 约束到 AGENTS.md
- 新增架构模式到 ARCHITEACHTURE.md
- 创建新的 docs/ 下的主题文档

**R (Read) - 读取：**
- 加载现有知识地图内容
- 分析项目代码中的 harness 相关模式

**U (Update) - 更新：**
- 更新 AGENTS.md 中的 agent 行为规范
- 更新 ARCHITEACHTURE.md 中的架构描述
- 更新 docs/ 下的技术文档

**D (Delete) - 删除：**
- 移除过时的知识地图条目
- 清理不再适用的约束

## 知识地图三层结构

```
┌─────────────────────────────────────────┐
│               知识地图                    │
├─────────────────────────────────────────┤
│  约束层 (Constraints)                   │
│  - 项目编码规范                         │
│  - Git 工作流约束                       │
│  - 工具使用限制                         │
├─────────────────────────────────────────┤
│  反馈层 (Feedback)                      │
│  - 即时反馈机制                        │
│  - 延迟反馈通道                        │
│  - 验证标准                            │
├─────────────────────────────────────────┤
│  控制层 (Control)                       │
│  - 任务状态管理                        │
│  - 上下文管理策略                      │
│  - 资源限制                            │
└─────────────────────────────────────────┘
```

```markdown
## 架构概览
（系统组件图）

```

### docs/{topic}.md
特定主题的详细文档

---

## 执行原则

1. **增量更新**：保留现有内容，只添加或修改必要的部分
2. **基于证据**：从项目代码中提取真实模式，不编造
3. **可执行约束**：约束应该具体可执行，不是模糊的建议
4. **版本追踪**：记录每次更新的内容和时间
