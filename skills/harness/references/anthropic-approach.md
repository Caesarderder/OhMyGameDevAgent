# Anthropic Harness Engineering

## 官方定义

Anthropic 官方文档定义：

> **"An agent harness (or scaffold) is the system that enables a model to act as an agent: it processes inputs, orchestrates tool calls, and returns results."**
>
> — Anthropic, "Demystifying evals for AI agents"

---

## Anthropic 的核心观点

### 1. Harness 是运行时操作系统

```
模型 (Model) + Harness (操作系统) = Agent
     ↓              ↓
   CPU核心      完整的运行时
```

Harness 负责：
- 输入处理
- 工具调用编排
- 结果返回
- 状态管理
- 错误恢复

### 2. 长时运行的挑战

Anthropic 工程师在博客中指出：

> "典型失败模式：在长时间运行的 Agent 中，仅有上下文压缩（compaction）远远不够。"

**需要解决的问题**：
- 上下文耗尽
- 错误累积
- 状态漂移
- 目标遗忘

## Claude Code 架构解析

Claude Code 泄露源码揭示的六大核心组件：

### 1. 多层级 System Prompt

```
┌─────────────────────────────────────────────────────┐
│  固定缓存部分（可缓存、十几万token）                  │
│  ├── Agent 身份定义                                 │
│  ├── Co 指令（行为规范）                            │
│  ├── 工具定义                                       │
│  ├── 语气规范                                       │
│  └── 安全策略                                       │
├─────────────────────────────────────────────────────┤
│  动态可替换部分（每次会话不同）                      │
│  ├── 会话状态                                       │
│  ├── 当前时间                                       │
│  ├── 可读取文件列表                                  │
│  └── 代码包依赖信息                                  │
└─────────────────────────────────────────────────────┘
```

**设计优势**：
- 固定部分只需加载一次，可缓存
- 动态部分实现个性化
- 模型注意力负担更低

### 2. Tool Schema

**影响**：直接决定工具调用准确率

**设计原则**：
- 内置核心工具在模型训练阶段完成适配
- 企业级场景：拒绝无权限校验的第三方工具
- 工具描述要精确，避免歧义

### 3. ToolCallLoop

Harness 最核心部分：

```
         ┌──────────────────────────────────────┐
         │                                      │
    ┌────▼────┐    ┌───────────┐    ┌──────┐   │
    │  Plan   │───▶│   Act     │───▶│Observe│   │
    │ (规划)  │    │  (执行)   │    │(观察)│   │
    └─────────┘    └───────────┘    └──────┘   │
         ▲                                 │    │
         │                                 │    │
         └─────────────────────────────────┘    │
                      (反馈循环)                   │
                                                   │
    ┌─────────┐    ┌───────────┐    ┌──────┐    │
    │ Result  │◀───│ Evaluate  │◀───│Loop  │────┘
    │ (结果)  │    │  (评估)    │    │检测  │
    └─────────┘    └───────────┘    └──────┘
```

**规划模式**：
1. 理解任务
2. 梳理文件系统
3. 明确可用工具
4. 生成执行方案

**执行模式**：
1. 在沙盒中按规划执行
2. 获取结果闭环
3. 错误检测与恢复

### 4. ContextManager

**核心问题**：如何高效利用百万级 token 上下文？

**解决方案**：
- 指针索引式 Memory
- 上下文压缩（compaction）
- 重要性排序


## Generator-Evaluator 模式

### 为什么需要分离？

Anthropic 工程师通过借鉴 GAN 的思想发现问题：

> "AI 自我评估过于宽容——它不会严格要求自己。"

**解决**：将执行工作的代理与评判工作的代理分开。

### 前端设计任务案例

**定义四个可量化标准**：
1. 设计质量（美学、专业程度）
2. 原创性（创新程度）
3. 工艺（细节、完善度）
4. 功能性（可用性、用户体验）

**迭代流程**：
```
Generator → Evaluator → 反馈 → Generator → ...
     ↑                                    │
     └────────────────────────────────────┘
                    (多轮迭代)
```

**结果**：在主观性极强的前端设计中，AI 展现出了"博物馆级"的审美判断。

## Context Reset 策略

### 问题

长时运行 agent 会遇到：
- 上下文耗尽
- 早期信息被遗忘
- 累积错误

### 解决方案

分段交接，每个阶段有明确的上下文边界：

```
┌─────────────┐    交接棒    ┌─────────────┐
│   Stage 1   │ ──────────→ │   Stage 2   │
│  0-50k tokens             │  0-50k tokens (新)
│  完整上下文  │   传递摘要   │  从 Stage 1  │
└─────────────┘   和指针    │  继承摘要    │
                                  │
                                  ▼
                           ┌─────────────┐
                           │   Stage 3   │
                           │  0-50k tokens
                           │  从 Stage 2 │
                           └─────────────┘
```

**关键设计**：
- 通过指针索引而非完整上下文传递状态
- 每个阶段独立上下文窗口
- 降低幻觉风险

## 全栈开发案例

Anthropic 工程师展示的 6 小时自主构建完整应用：

### 架构

```
┌─────────┐    ┌────────────┐    ┌───────────┐
│ Planner │ →  │  Generator  │ →  │  Evaluator │
│ 规划器  │    │   生成器    │    │   评估器   │
└─────────┘    └────────────┘    └───────────┘
     │                                    │
     │           反馈循环                  │
     └────────────────────────────────────┘
```

### 冲刺合同机制

每个冲刺有明确目标：
- 目标：完成用户认证模块
- 验收标准：注册、登录、登出流程可测试
- 时长：45 分钟

### 关键洞察

> "随着模型能力提升（Opus 4.5→4.6），框架可以简化，但评估器在任务处于模型能力边界时仍然关键。"

## 企业级实践

### 可靠性保障

```yaml
verification:
  pre_deploy:
    - 单元测试通过
    - 集成测试通过
    - 代码规范检查
    - 安全扫描
  
  continuous:
    - 错误率监控
    - 延迟监控
    - 资源使用监控
```

### 权限与安全

- 工具调用需要显式授权
- 敏感操作需要二次确认
- 操作日志完整记录
