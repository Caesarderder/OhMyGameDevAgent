## 案例 1: OpenAI 100 万行代码项目

### 项目背景

**时间**：2026 年初
**团队**：3 名工程师（后扩展至 7 人）
**条件**：完全禁止手写代码
**成果**：100 万行代码、完整产品内测版

### 关键成功因素

1. **完善的 Harness 系统**
   - 自动化代码审查
   - CI/CD 管道
   - 错误自动分类

2. **冲刺合同机制**
   - 每个冲刺 1-2 周
   - 明确目标和验收标准
   - 每日 standup

3. **人机协作模式**
   - AI 负责实现
   - 人类负责验收和方向把控

### 经验教训

| 教训 | 应用 |
|------|------|
| 不要让 Agent 自由发挥 | 设计护栏和边界 |
| 频繁验证 | 不要等最后才检查 |
| 记录错误模式 | 系统性修复而非个体修复 |

## 案例 2: Anthropic 前端设计实验

### 实验背景

Anthropic 工程师设计实验：
- **任务**：让 AI 自主完成前端设计
- **挑战**：设计是主观的，没有标准答案
- **方法**：Generator-Evaluator 分离

### 四维评估标准

1. **设计质量**：美学、专业程度
2. **原创性**：创新程度、差异化
3. **工艺**：细节、完善度
4. **功能性**：可用性、用户体验

### 迭代流程

```
Generator → 产出设计 → Evaluator → 反馈 → Generator → ...
     ↑                                              │
     └──────────────────────────────────────────────┘
                        (多轮迭代)
```

### 成果

> "在前端设计任务中，通过多轮迭代，成功让 Claude 产出了具有创造性飞跃的设计，展现出了'博物馆级'的审美判断。"

## 案例 4: 独立开发者 2D 游戏制作器

### 对比实验

**纯 Prompt 方法**：
- 耗时：10 分钟
- 花费：9 美元
- 结果：基础版本，很多 bug

**Harness Engineering 方法**：
- 耗时：6 小时
- 花费：~200 美元
- 结果：完整、可玩、设计精美

### 关键差异

```
纯 Prompt：
User → AI → Output (无反馈循环)

Harness Engineering：
User → Planner → Generator → Evaluator → Feedback → ...
                ↑                              │
                └──────────────────────────────┘
                        (持续迭代)
```

## 案例 5: Claude Code 架构解析

### 泄露事件

2026 年 3 月底，Claude Code 源码泄露：
- **1987 个 TypeScript 文件**
- **51.2 万行代码**
- 完整工程实现曝光

### 六大核心组件

| 组件 | 功能 |
|------|------|
| Multi-level System Prompt | 分层缓存、动态替换 |
| Tool Schema | 工具定义与权限 |
| ToolCallLoop | 规划+执行双模式 |
| ContextManager | 指针索引式 Memory |
| SubAgent | 任务分解与协作 |
| VerificationHooks | 自动化质量检查 |

### 架构优势

1. **简洁的 System Prompt**：模型注意力负担低
2. **明确的工具边界**：调用准确率高
3. **健壮的状态管理**：支持长时运行

## 案例 7: 企业级合规文档查询 Agent

### 项目要求

- 24/7 自动化客服
- 合规性保证
- 审计追踪

### 七层架构

```
┌─────────────────────────────────────┐
│           任务层                    │ ← 用户请求解析
├─────────────────────────────────────┤
│          上下文层                   │ ← RAG 检索
├─────────────────────────────────────┤
│           工具层                   │ ← 文档查询
├─────────────────────────────────────┤
│          规则层                    │ ← 合规校验
├─────────────────────────────────────┤
│          验证层                    │ ← 答案验证
├─────────────────────────────────────┤
│          观测层                    │ ← 监控告警
├─────────────────────────────────────┤
│         反馈闭环层                  │ ← 持续改进
└─────────────────────────────────────┘
```

### 关键设计

1. **规则引擎**：强制合规检查
2. **答案溯源**：每个答案有引用
3. **人工复核**：高风险答案需确认

## Mitchell Hashimoto 的六步进阶

来自 HashiCorp 联合创始人的经验：

### Step 1: 最小可行 Harness

从一个能跑起来的简单循环开始。

### Step 2: 添加错误处理

捕获错误，记录模式，添加恢复。

### Step 3: 实现状态持久化

支持中断和恢复。

### Step 4: 引入验证层

不要相信 Agent 输出。

### Step 5: 优化上下文管理

实现压缩和摘要。

### Step 6: 建立反馈闭环

系统性学习和改进。
