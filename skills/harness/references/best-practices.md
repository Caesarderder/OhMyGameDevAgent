# Harness Engineering 最佳实践

---

## 设计原则

### 1. 每次犯错，修复系统而非责怪模型

```
Agent 犯错
    │
    ▼
分析错误模式
    │
    ├── 是模型能力问题？ → 考虑换模型或调整提示
    │
    └── 是系统设计问题？ → 修复 Harness
                              │
                              ▼
                         更新系统
                         不再犯同类错误
```

### 2. 让 Agent 学会"知道自己不知道"

**自我评估提示词设计**：

```markdown
在执行任务前，请先评估：
1. 你是否理解任务的全部要求？
2. 你是否具备完成任务所需的工具？
3. 你有多大把握能成功？（1-10）
4. 如果不确定，哪些部分需要澄清？

如果信心低于 7 分，请先提问而非盲目执行。
```

### 3. 设计高速公路，而非控制每辆车

**类比**：
- Prompt Engineering = 教某人如何开车
- Harness Engineering = 建造一条有护栏、限速、加油站的高速公路

**实践**：
- 设置边界（护栏）
- 提供清晰的方向（路标）
- 实现错误恢复（应急车道）
- 建立反馈机制（监控摄像头）

### 4. 拥抱渐进式复杂度

```
阶段 1: 单工具调用
    ↓
阶段 2: 多工具顺序调用
    ↓
阶段 3: 条件分支和循环
    ↓
阶段 4: 多 Agent 协作
```

每个阶段验证通过后再进入下一阶段。

### 5. 测量一切可测量的

```yaml
metrics:
  任务完成率: true
  平均执行时间: true
  错误率: true
  工具调用成功率: true
  上下文利用率: true
  自我纠错率: true
```

## 架构模式

### 模式 1: Planner-Generator-Evaluator

```
┌─────────────┐
│   Planner   │ ← 分解任务、制定计划
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Generator  │ ← 执行具体工作
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Evaluator  │ ← 评估结果、提供反馈
└──────┬──────┘
       │
       │ (循环直到达标)
       ▼
```

**适用场景**：复杂任务、需要严格质量把控

### 模式 2: Supervisor-Worker

```
┌────────────┐
│ Supervisor │ ← 负责任务分配、结果汇总
└─────┬──────┘
      │
      ├──→ Worker A
      ├──→ Worker B
      └──→ Worker C
           (并行执行子任务)
```

**适用场景**：可并行的独立子任务

### 模式 3: State Machine

```
     ┌──────────────────────────────────────┐
     ▼                                      │
┌─────────┐    ┌─────────┐    ┌─────────┐ │
│  IDLE   │ → │ RUNNING │ → │  DONE   │ │
└─────────┘    └────┬────┘    └─────────┘ │
                     │                      │
                     ▼                      │
                ┌─────────┐                 │
                │ BLOCKED │ ← 等待外部输入  │
                └─────────┘                 │
                     │                      │
                     ▼                      │
                ┌─────────┐                 │
                │  ERROR  │ → 错误恢复      │
                └─────────┘                 │
```

**适用场景**：有明确状态转换的任务

## 工具设计

### 好的工具定义

```json
{
  "name": "search_code",
  "description": "Search for code patterns in the codebase",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "Search query (supports regex)"
      },
      "file_pattern": {
        "type": "string",
        "description": "File pattern to search (e.g., *.js)"
      },
      "max_results": {
        "type": "integer",
        "description": "Maximum results to return",
        "default": 50
      }
    },
    "required": ["query"]
  }
}
```

### 工具设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 单一职责 | 每个工具只做一件事 | `read_file` 而非 `read_and_analyze_file` |
| 明确输入 | 参数定义清晰 | 不要 `options` 对象 |
| 原子操作 | 不可拆分的基本操作 | `write_file` 而非 `create_project` |
| 幂等性 | 多次执行结果一致 | `read_file` 是幂等的 |
| 错误处理 | 失败时返回明确信息 | 包含错误码和消息 |

### 避免的工具设计

❌ **上帝工具**：一个工具做所有事
```javascript
// 避免：工具过于复杂
{
  "name": "do_everything",
  "description": "Can do any task you ask",
  // ...
}
```

✅ **聚焦工具**：每个工具职责单一
```javascript
// 推荐：职责明确
{ "name": "read_file", ... }
{ "name": "write_file", ... }
{ "name": "list_directory", ... }
```

## 错误处理

### 错误分类

```yaml
error_types:
  recoverable:
    - 网络超时（重试）
    - 文件不存在（创建）
    - 权限不足（申请）
  
  unrecoverable:
    - 语法错误（需要人工修复代码）
    - 循环依赖（需要重新设计）
    - 资源耗尽（需要扩容）
```

### 错误恢复策略

```
错误发生
    │
    ▼
错误分类
    │
    ├── 可恢复？ ──是──→ 自动重试
    │      │
    │      └──否──→ 降级处理
    │                    │
    └── 人工介入？ ──是──→ 暂停、告警
           │
           └──否──→ 记录、跳过
```

### 断点恢复

```javascript
// 状态保存
{
  "checkpoint": {
    "task_id": "task_123",
    "step": 5,
    "completed_steps": [1, 2, 3, 4],
    "current_state": {...},
    "timestamp": "2026-04-21T10:30:00Z"
  }
}

// 恢复时从 step 5 继续
```

## 上下文管理

### 压缩策略

```javascript
// 上下文压缩
{
  "original_length": "100k tokens",
  "compression": {
    "strategy": "importance_weighted",
    "保留": [
      "当前任务目标",      // 高权重
      "最近的工具调用结果", // 高权重
      "错误日志",          // 高权重
      "历史上下文摘要"     // 中权重
    ],
    "丢弃": [
      "早期的成功步骤",
      "已验证正确的代码段"
    ],
    "compressed_length": "30k tokens"
  }
}
```

```
┌─────────────────────────────────────┐
│          工作记忆 (Working)          │ ← 当前任务
├─────────────────────────────────────┤
│          短期记忆 (Short-term)      │ ← 最近 10 分钟
├─────────────────────────────────────┤
│          长期记忆 (Long-term)       │ ← 项目知识
└─────────────────────────────────────┘
```

## 性能优化

### 工具调用优化

1. **并行化**：独立工具可并行调用
2. **缓存**：重复查询返回缓存结果
3. **批处理**：多个小操作合并为批量操作
4. **预取**：预测下一个需要的工具

### 上下文优化

1. **延迟加载**：不立即需要的信息后加载
2. **选择性检索**：只检索相关内容
3. **摘要压缩**：长文本用摘要替代
4. **指针共享**：多个地方引用同一数据
