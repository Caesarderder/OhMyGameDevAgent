# OpenAI Harness Engineering

## 官方定义

2025 年 2 月 11 日，OpenAI 发布报告：

> **"Harness engineering: leveraging Codex in an agent-first world"**

Harness Engineering 是一种系统性方法，通过构建约束、工具、文档和反馈循环，使 AI 编码 Agent 能够可靠地完成工作。

---

## OpenAI 的核心观点

### 1. 模型只是 CPU

> "The model is the CPU, and the harness is the operating system."

模型本身不产生价值——只有配上完整的运行时系统，才能真正交付成果。

### 2. 工具定义决定一切

Codex 的工具系统设计原则：

```python
# 好的工具定义
{
    "name": "read_file",
    "description": "Read contents of a file",
    "parameters": {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "Absolute path to file"}
        },
        "required": ["path"]
    }
}

# 避免过度复杂化
# 保持参数简单、明确
```

### 3. 反馈循环是核心

```
┌──────────┐    错误模式    ┌──────────┐
│  Agent   │ ───────────→  │  System  │
│  犯错     │ ←───────────  │  修复     │
└──────────┘    解决方案    └──────────┘
```

## OpenAI 案例：100 万行代码项目

### 项目背景

2026 年初，OpenAI 团队实验：
- **3 名工程师**（后扩展至 7 人）
- **5 个月**时间
- **0 行手写代码**
- **100 万行代码**交付

### 关键成功因素

1. **完善的 Harness 系统**
   - 自动化代码审查
   - 持续集成/部署
   - 错误自动分类

2. **明确的验收标准**
   - 每个功能有可测试的验收条件
   - 自动化测试覆盖

3. **迭代反馈机制**
   - 每日 standup（AI + 人类）
   - 快速识别和修复问题

### 经验教训

| 教训 | 应用 |
|------|------|
| 不要让 Agent 自由发挥 | 设计护栏和边界 |
| 频繁验证 | 不要等最后才检查 |
| 记录错误模式 | 系统性修复而非个体修复 |

## OpenAI 推荐的 Harness 组件

### 1. 任务管理系统

```
任务输入 → 任务分解 → 优先级排序 → 执行 → 验证 → 交付
```

### 2. 沙盒执行环境

- 隔离运行，防止错误代码影响系统
- 可重现的构建环境
- 资源限制和监控

### 3. 自动化测试框架

```yaml
# 典型的测试金字塔
├── 单元测试 (快速、频繁)
├── 集成测试 (中等速度)
├── 端到端测试 (慢速、少次)
└── 人工审查 (关键节点)
```

### 4. 错误追踪系统

- 自动分类错误类型
- 关联相似错误
- 建议修复方案

### 5. 文档生成系统

- 代码即文档
- 自动更新依赖关系
- API 文档同步

## Codex 的工具调用策略

### 工具选择原则

1. **能力覆盖**：工具集要覆盖 Agent 所需能力
2. **使用复杂度**：工具越简单越好
3. **调用时机**：避免不必要的调用

### 避免的陷阱

❌ **工具过多**：Agent 难以选择
```
应该：10-15 个核心工具
避免：100+ 个工具
```

❌ **工具过于相似**：选择困难
```
应该：功能明确区分
避免：多个工具做类似事情
```

❌ **缺乏错误处理**：单点失败
```
应该：每个工具有错误处理
避免：工具调用没有 try/catch
```
