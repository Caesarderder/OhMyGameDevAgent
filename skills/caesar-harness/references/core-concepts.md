# Harness Engineering 核心概念

## 定义

**Harness Engineering（驾驭工程）** 是系统性地构建约束、工具、文档和反馈循环的学科，使 AI 编码 Agent 能够可靠地完成工作。

> 核心原则：*每次发现 Agent 犯错，就花时间设计一个解决方案，确保它不再犯同样的错误。*
> — Mitchell Hashimoto, HashiCorp 联合创始人

---

## 核心公式

```
Agent = LLM + Harness
```

Harness 是除模型本身外，所有决定系统能否稳定交付的组件总和。

---

## 三阶段演进

### Stage 1: Prompt Engineering (2023)

**核心问题**：模型是否听懂了指令？

**解决思路**：优化语言表达
- 角色设定
- 风格约束
- 示例引导

**局限性**：
- 无法弥补知识缺失
- 不能无限制地提供信息

### Stage 2: Context Engineering (2025)

**核心问题**：模型是否获得了正确的信息？

**解决思路**：优化信息供给
- RAG（检索增强生成）
- 渐进式披露
- 信息分层

**局限性**：
- 无法解决执行过程中的监督与纠错问题

### Stage 3: Harness Engineering (2026)

**核心问题**：模型能否稳定执行任务？

**解决思路**：优化运行系统
- 执行编排
- 状态管理
- 错误恢复

**地位**：包含前两者，是更高层级的系统工程

## 六层架构体系

```
┌─────────────────────────────────────┐
│         上下文边界层                 │ ← 确保模型在正确的边界内思考
├─────────────────────────────────────┤
│          工具系统层                 │ ← 连接模型与现实世界的桥梁
├─────────────────────────────────────┤
│          编排控制层                 │ ← 规划、决策、执行流程
├─────────────────────────────────────┤
│          状态管理层                 │ ← 记忆、连续性、上下文
├─────────────────────────────────────┤
│          验证评估层                 │ ← 质量门禁、错误检测
├─────────────────────────────────────┤
│          反馈闭环层                 │ ← 自我纠错、持续改进
└─────────────────────────────────────┘
```

## 关键概念详解

### Context Window vs Context Management

- **Context Window**：模型能处理的 token 数量（有限资源）
- **Context Management**：如何高效利用有限上下文（管理策略）

### Tool Schema Design

工具定义直接影响模型调用准确率。设计原则：
1. **清晰**：每个工具有明确的输入输出
2. **简洁**：避免过度参数化
3. **一致**：命名和格式保持统一
4. **可测试**：每个工具可独立验证

### Agent Loop 模式

```
while task_not_complete:
    1. Think (规划下一步)
    2. Act (执行工具调用)
    3. Observe (获取结果)
    4. Evaluate (评估结果)
    5. Adapt (调整策略)
```

### Checkpoint & Resume

长时任务需要定期保存状态：
- 当前进度
- 中间结果
- 上下文摘要
- 错误记录
